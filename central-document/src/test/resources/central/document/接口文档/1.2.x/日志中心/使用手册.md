## 概述
日志中心主要用于采集、过滤、存储所有微服务产生的日志信息。日志中心已经提供了相关的过滤、存储手段，因此只需要各个微服务将日志按指定方式传输到日志采集器即可。

日志中心通过插件机制，可以提供多种日志入口。目前日志中心已提供了 http 形式接入和 Rabbit 形式的接入（Kafka 的接入还处理规划中）。

## 日志格式

| 参数名 | 类型 | 必填 | 默认值 | 说明 |

## 日志传输
### HTTP 接口传输
如果业务系统决定使用 HTTP 接口的方式传输日志，那么可以将日志通过 HTTP 请求的方式传输到指定的采集器上即可。接口声明如下:

#### 接口地址
- `POST lb://mcube-logging/logging/api/collect/${path}`

> `${path}` 需要参数采集器的配置

#### 请求方法
- POST

#### Body 参数（application/json）
- `List<Log>`

需要注意的是，如果业务系统采用了 HTTP 接口的方式传输日志，一定要设计日志缓冲机制，等待日志收集到一定量或一定时间后，再向日起中心发起请求。这样可以有效降低客户端、日志中心的请求压力。

#### 实现示列

```java
// 日志队列
// 业务系统产生的所有日志，需要放到此队列里
private BlockedQueue<Log> queue;

// Http Client
private HttpClient client;

// 另起一个线程池，用于消费队列里的日志
public void run() {
    try {
        while( true ) {
            // 从队列里面取日志
            // 队列里超过 5000 条日志，或 1 秒之后，取出里面的日志进行收集
            // 这样既能保证日志不会发送过于频繁，也能保证日志能够及时收集
            List<Log> logs = queue.poll(5000, 1000, TimeUnit.MILLISECONDS);
            if (logs.isEmpty()){
                // 1 秒内没有产生日志
                continue;
            }
            
            // 使用 Gzip 压缩日志消息，可以减少日志对网络带宽的占用
            // 据测试，使用 Gzip 压缩后的日志比原来的日志减少了 90% 以上的流量
            ByteArrayOutputStream output = new ByteArrayOutputStream();
            try (GZIPOutputStream stream = new GZIPOutputStream(output)){
                stream.write(MCubeJSON.Default().serializeAsBytes(logs), false));

                // 由于后续存在耗时操作，因此需要将引用置空，这样 GC 可以更快释放对象
                logs = null;
            } catch (IOException ignored){}
            
            // 使用 Http Client 发送消息体
            // 注意，如果使用了 GZip 压缩了消息体，则发送请求时需要将请求头 Content-Encoding 指定为 gzip
            client.post(new ByteArrayInputStream(output.toByteArray()));
        }
    } catch(InterruptedException interrupt){
        Thread.currentThread().interrupt();
    }
}
```

> 以上代码没有提供发送失败重试机制，这个需要开发人员自行处理。

### Rabbit 队列传输
如果业务系统决定使用消息队列（Rabbit）的方式传输日志，那么可将日志通过消息队列客户端进行发送。日志格式需要使用 `List<Log>` 序列化成 JSON 后发送到消息队列。

实施人员需要在日志中心添加 RabbitMQ 类型的日志采集器，并设置相同的队列、路由后，即可监听 RabbitMQ 消息队列，从队列中获取日志并进入过滤、存储逻辑。

开发人员在设计 RabbitMQ 发送机制时，可以设计一定的缓冲机制，等待日志满足一定数量和一定时间后，再来发送日志。下面提供一种简单的实现方案:

```java
// 日志队列
// 业务系统产生的所有日志，需要放到此队列里
private BlockedQueue<Log> queue;

// Rabbit 发送器
private RabbitTemplate template;

// 另起一个线程池，用于消费队列里的日志
public void run() {
    try {
        while( true ) {
            // 从队列里面取日志
            // 队列里超过 5000 条日志，或 1 秒之后，取出里面的日志进行收集
            // 这样既能保证日志不会发送过于频繁，也能保证日志能够及时收集
            List<Log> logs = queue.poll(5000, 1000, TimeUnit.MILLISECONDS);
            if (logs.isEmpty()){
                // 1 秒内没有产生日志
                continue;
            }
            
            // 使用 Gzip 压缩日志消息，可以减少日志对网络带宽的占用
            // 据测试，使用 Gzip 压缩后的日志比原来的日志减少了 90% 以上的流量
            ByteArrayOutputStream output = new ByteArrayOutputStream();
            try (GZIPOutputStream stream = new GZIPOutputStream(output)){
                stream.write(MCubeJSON.Default().serializeAsBytes(logs), false));

                // 由于后续存在耗时操作，因此需要将引用置空，这样 GC 可以更快释放对象
                logs = null;
            } catch (IOException ignored){}
            
            // 构建消息属性
            MessageProperties properties = new MessageProperties();
            properties.setDeliveryMode(MessageDeliveryMode.PERSISTENT);
            properties.setContentType("application/json");
            // 告诉接收方当前使用了 GZip 压缩
            properties.setContentEncoding("gzip");
            properties.setMessageId(MCubeGUID.nextID());
            
            // 消息路由键
            String routingKey = "mcube.logging";
            
            // 构建消息体
            Message message = new Message(output.toByteArray(), properties);
            // 发送消息
            template.send(exchange.getName(), routingKey, message);
        }
    } catch(InterruptedException interrupt){
        Thread.currentThread().interrupt();
    }
}
```

> 以上代码没有提供发送失败重试机制，这个需要开发人员自行处理。

### 其它方式日志传输
以上无论是 HTTP 传输，还是消息队列传输，都需要开发者去自定一个日志的 Appender 来完成日志发送的工作。如果开发者认为不想对现有的应用系统进行改变，或者希望采用被动式的日志采集来完成日志传输工作，那么开发者可以使用 Filebeat 方案进行日志采集和传输。

在这种方案下，微服务使用 LogBack 或 Log4j 这些日志框架，将日志直接写入本地文件。Filebeat 组件会监控这些日志文件，当日志文件发生变更时，将变更内容传输到 Logstash。Logstash 在收到日志数据后，需要将这些日志转化为日志中心要求的日志格式，再输出到 Rabbit 队列即可。日志中心的采集器会去监听消息队列，获得日志内容。各组件的交互图如下:
